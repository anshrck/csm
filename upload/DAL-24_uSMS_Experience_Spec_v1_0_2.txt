1. Purpose and Scope
────────────────────────────────────────────────────────────────────────────────

1.1 Purpose

This document specifies the Experience Layer of uSMS at two levels:

  Level 1 — Functional Space Definition
    Per workspace: which entities are visible, which actions are available,
    which status views exist, which notifications are surfaced.
    Answers: "What belongs in this workspace?"

  Level 2 — Screen Inventory & Interaction Model
    Per screen within each workspace: trigger conditions, data sources
    (from usms_ops and usms_catalog), available actions (state transitions),
    and role/entitlement constraints.
    Answers: "What happens on each screen?"

This document does not specify visual design, component libraries, or
CSS implementation. It specifies functional behavior and data contracts.
Visual design follows in a subsequent Component Model (DAL-23).

1.2 Scope — Six Workspaces

  WS-01  CSM Portal             Service Customer
  WS-02  CSM Workspace          SCM Worker, CM Leader
  WS-03  Service Desk Portal    Service User
  WS-04  SD Workspace           SD Agent, SD Leader
  WS-05  PM Workspace           PM Worker, PM Leader
  WS-06  CE & Config Workspace  CE Worker, CE Leader, Technical Owner

1.3 Governing Principles (inherited from DAL-20)

  Consume Never Own
    No workspace stores or owns domain data. All data is consumed from
    uSMS domain APIs (/ops/*) and uSeC catalog APIs (/catalog/*).

  Assist or Redirect
    Business operations (state transitions, approvals, closures) are
    executed by domain services — not by the Experience Layer.
    The workspace surfaces the action; the API enforces the rule.

  Real-Time by Default
    All workspace state is event-driven. Lists, counters, and status
    indicators update via Event Bus subscription — no polling.

  Entitlement-Governed Rendering
    Every screen, panel, and action button is rendered based on the
    caller's resolved entitlement and role. No UI configuration bypasses
    this. If the API would deny the action, the button does not appear.

  Reduced to the Max (HRX)
    No workspace surface adds cognitive load without governance value.
    Every element must justify its presence by enabling a decision,
    surfacing a status, or triggering a legitimate action.

1.4 Notation

  Screen ID:    WS-{workspace}-{sequence}  (e.g. WS-01-003)
  Data:         [Entity] from schema        (e.g. [Demand] from usms_ops)
  Actions:      → STATE                    (e.g. → QUOTED)
  Constraints:  role, scope, status gate   (e.g. SERVICE_CUSTOMER only)

2. WS-01 — CSM Portal  (Service Customer)
────────────────────────────────────────────────────────────────────────────────

2.1 Functional Space Definition

  Purpose
    The CSM Portal is the Service Customer's strategic interface with the
    service organization. It surfaces the customer's demand portfolio,
    active service commitments, SLA performance, and the Service Catalog
    available to their organizational scope.

  Role
    SERVICE_CUSTOMER — scoped to their OrgNode and children.

  Visible Entities
    [Demand]             Own demands only (serviceCustomerId = caller)
    [Service]            Services in usms_catalog entitled to their OrgNode
    [ServiceOffering]    Entitled offerings within their scope
    [SLAProfile]         Profiles attached to their active services
    [SlaEvent]           SLA performance events for their services
    [Entitlement]        Their OrgNode's entitlement records (read-only)

  Available Actions
    Submit new demand
    Accept or decline a quote
    Close a fulfilled or redirected demand
    View SLA performance per service

  Status Views
    Demand pipeline: NEW / UNDER_REVIEW / QUOTED / ACCEPTED / IN_CHANGE / FULFILLED
    SLA health: per active service (green / warning / breached)
    Service Catalog: available offerings within entitlement scope

  Notifications
    DemandQuoted          — quote ready for review
    DemandFulfilled       — demand completed
    DemandRejected        — demand not feasible (with reason)
    SlaWarning            — SLA at 70% threshold for an entitled service
    SlaBreached           — SLA breached for an entitled service
    ServiceCatalogUpdated — new service available within scope

  Excluded
    Incident records (those belong to SD Portal)
    Problem records (internal process — not customer-facing)
    Change records (internal — customer sees effect via Demand status)
    Other customers' data (strict OrgNode scope)

2.2 Screen Inventory — WS-01

────────────────────────────────────────────────────────────────────────────────
WS-01-001  Dashboard
────────────────────────────────────────────────────────────────────────────────

  Trigger
    Landing screen after authentication for SERVICE_CUSTOMER role.

  Purpose
    Single-glance view of demand pipeline, SLA health, and pending actions.

  Data
    [Demand]    Count by DemandStatus — own demands
    [SlaEvent]  Active SLA warnings and breaches for entitled services
    [Demand]    Demands awaiting customer action (status = QUOTED)

  Panels
    My Demand Pipeline
      Visual status lane: NEW → UNDER_REVIEW → QUOTED → IN_CHANGE → FULFILLED
      Click-through to WS-01-003 (Demand Detail)
      Count per lane. Real-time via event subscription.

    Pending My Action
      List of Demands in status = QUOTED requiring customer acceptance.
      Each row: Demand title, SCM Worker, quote date, estimated effort/cost.
      Action button: Review Quote → WS-01-004

    SLA Health
      Per active service: service name, SLA Class, current health indicator.
      Green (within target) / Amber (warning) / Red (breached).
      Click-through to WS-01-006 (SLA Detail)

    Recent Activity
      Last 10 domain events relevant to customer scope.
      DemandQuoted, DemandFulfilled, SlaWarning, ServiceCatalogUpdated.

  Actions
    [New Demand] button → WS-01-002
    [Review Quote] per quoted demand → WS-01-004

  Constraints
    All data scoped to caller's OrgNode (and children).
    No data from other customers rendered.

WS-01-002  Submit Demand
────────────────────────────────────────────────────────────────────────────────

  Trigger
    [New Demand] action from WS-01-001 or WS-01-003.

  Purpose
    Structured demand submission form. Creates Demand record in status NEW.

  Data
    [Service]          Service Catalog browse — entitled services as reference
    [ServiceOffering]  Available offerings to pre-populate context

  Form Fields
    Title              (required, free text)
    Description        (required, rich text — what is needed and why)
    Related Services   (optional multi-select from entitled catalog)
    Desired Timeline   (optional — informational for SCM Worker)
    Business Justification (optional — supports SCM Worker assessment)

  Actions
    [Submit Demand]
      POST /ops/demands
      Creates Demand (status → NEW)
      Emits DemandCreated
      Returns to WS-01-001 with confirmation

    [Cancel] — discard and return to Dashboard

  Constraints
    SERVICE_CUSTOMER role required (BR-CM-001).
    Cannot submit demand on behalf of another OrgNode.
    Title and Description are mandatory — form blocks submission if empty.

WS-01-004  Demand Detail
────────────────────────────────────────────────────────────────────────────────

  Trigger
    Row click from WS-01-003, or notification deep-link (DemandQuoted).

  Purpose
    Full demand record view. Primary action surface for quote acceptance.

  Data
    [Demand]           Full record including all fields

WS-01-006  SLA Performance
────────────────────────────────────────────────────────────────────────────────

  Trigger
    SLA Health panel click from WS-01-001, or top navigation.

  Purpose
    Per-service SLA performance overview for the Service Customer.

  Constraints
    Read-only for Service Customer.
    Incident details are not exposed — only aggregate SLA metrics.

3. WS-02 — CSM Workspace  (SCM Worker, CM Leader)
────────────────────────────────────────────────────────────────────────────────

3.1 Functional Space Definition

  Purpose
    The CSM Workspace is the operational environment for SCM Workers and
    CM Leaders. It manages the full demand lifecycle, supports SLM governance,
    and provides the interface for catalog-related customer communications.

  Roles
    SCM_WORKER   — primary operator
    CM_LEADER    — oversight, throughput monitoring, escalation

  Visible Entities
    [Demand]           All demands within tenant scope
    [Service]          Full Service Catalog (usms_catalog)
    [ServiceOffering]  All offerings
    [SLAProfile]       All profiles
    [SlaEvent]         All SLA events
    [Change]           Linked changes (status view only)
    [ProcessHandover]  CM_TO_CE handovers created by this workspace

  Available Actions
    Assign demand to SCM Worker
    Assess demand (catalog check, feasibility)
    Redirect or reject demand
    Create quote (effort, cost, timeline)
    Create Change Request and hand to CE
    Mark demand fulfilled / closed
    Manage SLA profiles and customer SLM reporting

  Status Views
    Demand queue by status
    Unassigned demands
    Demands awaiting customer action (QUOTED)
    SLA compliance per Service Customer

  Notifications
    DemandCreated         — new demand requires assignment
    DemandAccepted        — customer accepted quote → trigger Change Request
    SlaBreached           — breach requires SCM Worker customer communication
    ChangeClosed          — linked change complete → demand can be FULFILLED

WS-02-003  Demand Detail & Work Surface
────────────────────────────────────────────────────────────────────────────────

  Trigger
    Row click from WS-02-002, or notification deep-link.

  Purpose
    Full demand work surface for SCM Worker. All lifecycle actions available
    based on current status.

  Data
    [Demand]   Full record
    [Service]  Catalog lookup panel (side panel for assessment)
    [Change]   Linked change record (if exists)
    [User]     Service Customer contact info

  Sections
    Demand Header      Title, customer, submitted date, assigned worker
    Status Timeline    Visual state machine — current position highlighted
    Customer Input     Description and business justification (read-only)
    Catalog Check Panel
      Side panel: search Service Catalog while in UNDER_REVIEW.
      Confirms: "Is this demand already covered?"
      [Mark as Covered → Redirect] or [Proceed to Quote]
    Assessment Fields  Editable when status = UNDER_REVIEW:
      estimatedEffortDays / estimatedCost / quoteNotes
    Commitment Section Visible when status = ACCEPTED+:
      commitmentNotes / changeRequestId link
    Activity Log       Full event history with actor and timestamp

  Actions (visible by status)

    Status = NEW
      [Assign to Me]        PATCH /ops/demands/:id  (set assignedScmWorkerId)
      [Start Review]        POST /ops/demands/:id/review  → UNDER_REVIEW

    Status = UNDER_REVIEW
      [Redirect Demand]     POST /ops/demands/:id/redirect
                            Requires: redirectedToOfferingId selected in catalog panel
      [Reject Demand]       POST /ops/demands/:id/reject
                            Requires: rejectionReason text (dialog)
      [Submit Quote]        POST /ops/demands/:id/quote  → QUOTED
                            Requires: estimatedEffortDays, quoteNotes

    Status = ACCEPTED (customer accepted quote)
      [Create Change Request]
                            POST /ops/changes  (originType = DEMAND, originDemandId)
                            Then: POST /ops/demands/:id/hand-to-ce  → IN_CHANGE
                            Creates ProcessHandover (CM_TO_CE)

    Status = IN_CHANGE (change closed by CE)
      [Mark Fulfilled]      POST /ops/demands/:id/close  → FULFILLED → CLOSED
                            Only available when linked Change.status = CLOSED

  Constraints
    SCM_WORKER may only action demands assigned to them (or unassigned).
    CM_LEADER may action any demand in tenant.
    Quote submission validates estimatedEffortDays IS NOT NULL (BR-CM-002).

4. WS-03 — Service Desk Portal  (Service User)

4.1 Functional Space Definition

  Purpose
    The Service Desk Portal is the Service User's operational interface.
    It is the only channel through which Service Users interact with the
    service organization. It surfaces the user's active tickets, the
    available service catalog, and self-service options.

    SERVICE_USER — scoped to their OrgNode's entitlement set.

  Available Actions
    Report an incident
    Submit a service request
    Request access
    View and update own tickets
    Search Known Error database (self-service resolution)
    Confirm incident resolution

  Status Views
    My open incidents
    My open requests
    My resolved items (last 30 days)

  Notifications
    IncidentAssigned       — your incident has been picked up
    IncidentResolved       — your incident is resolved, confirmation requested
    ServiceRequestFulfilled — your request has been fulfilled
    ServiceRequestRejected — request rejected with reason
    KnownErrorMatchFound   — a Known Error was found matching your incident

  Excluded
    Other users' tickets
    Demand records (submit demand → contact Service Customer / your manager)
    Problem and Change records (internal)
    SLA metrics beyond own ticket status

────────────────────────────────────────────────────────────────────────────────
WS-03-002  Report Incident
────────────────────────────────────────────────────────────────────────────────

  Trigger
    [Report an Issue] from WS-03-001.

  Purpose
    Structured incident submission. Before form is shown, Known Error
    database is checked — if a match exists, it is surfaced first.

  Flow
    Step 1: Service Selection
      "Which service is affected?"
      Dropdown of entitled services (from entitlement resolution).
      On selection: background Known Error check fires.

    Step 2: Known Error Check (conditional)
      If active KnownError found for selected service:
        Display: Known Error title, workaroundDescription.
        [This resolves my issue] → no incident created, session ends.
        [The workaround doesn't help] → continue to Step 3.

    Step 3: Incident Form
      Title         (required)
      Description   (required — "what happened, when, what is the impact?")
      Priority hint (informational — user suggests, SD Agent confirms)
      Attachments   (optional)

    Step 4: Submission
      POST /ops/incidents
      Incident created (status → NEW)
      Emits IncidentCreated
      Confirmation screen with incident ID and reference number.

  Constraints
    serviceId required (BR-SD-001).
    Service must be in ACTIVE status and entitled to caller's OrgNode.

5. WS-04 — SD Workspace  (SD Agent, SD Leader)
────────────────────────────────────────────────────────────────────────────────

5.1 Functional Space Definition

  Purpose
    The SD Workspace is the operational environment for SD Agents and
    SD Leaders. It manages all active incidents, service requests, access
    operations, and event triage. It is the highest-throughput workspace
    in uSMS — designed for speed and decision clarity.

  Roles
    SD_AGENT   — primary operator (handles assigned tickets)
    SD_LEADER  — oversight (all tickets), SLA management, escalation authority

  Visible Entities
    [Incident]       All incidents in tenant
    [ServiceRequest] All service requests in tenant
    [KnownError]     All active Known Errors (KEDB)
    [Problem]        Read-only linkage (incident → problem)
    [SlaEvent]       All SLA events
    [Service]        Catalog lookup for CI identification
    [Entitlement]    For entitlement verification on requests

  Available Actions
    Assign incident or request to self or another agent
    Progress incident through state machine
    Escalate to Technical Owner or specialized group
    Apply Known Error workaround
    Verify entitlement and fulfill service request
    Reject service request

    Free-text field: approvalNotes — shared across board participants.
    Saved to Change record on board approval.

WS-06-006  CE Analytics
────────────────────────────────────────────────────────────────────────────────

  Trigger
    Navigation from top menu (CE_LEADER primary audience).

  Purpose
    Change throughput, quality, and governance health monitoring.

  Panels
    Change Throughput
      Changes opened / closed per period.
      Average lead time by type (STANDARD / NORMAL / EMERGENCY).

    Complexity Distribution
      SIMPLE / MEDIUM / COMPLEX ratio over time.
      Trend: is complexity growing? (governance signal)

    Change Origin Analysis
      Breakdown: DEMAND-origin vs PROBLEM-origin vs STANDARD.
      High PROBLEM-origin ratio → structural instability signal.

    Emergency Change Rate
      Emergency changes as % of total. Industry reference: <5%.
      Post-review completion rate.

    Technical Owner Activity
      Per Technical Owner: tasks assigned / completed / average completion time.
      Identifies overloaded or unresponsive Technical Owners.

    Failed Changes
      REJECTED changes with reason analysis.
      "Why are changes being rejected?" → governance quality signal.

8. Cross-Workspace Elements
────────────────────────────────────────────────────────────────────────────────

8.1 Global Navigation

  All authenticated users see:
    Logo / product name
    Workspace switcher (only workspaces for which they have a role)
    Notification bell (real-time, role-scoped)
    User profile / settings
    AI Assistant entry point (when Stage ≥ AI Assistant)

  Workspace switcher renders only entitled workspaces:
    SERVICE_CUSTOMER     → WS-01
    SCM_WORKER, CM_LEADER → WS-02
    SERVICE_USER         → WS-03
    SD_AGENT, SD_LEADER  → WS-04
    PM_WORKER, PM_LEADER → WS-05
    CE_WORKER, CE_LEADER, TECHNICAL_OWNER → WS-06
    Users with multiple roles see multiple workspaces.

8.2 Notification System

  Notifications are emitted by the Event Bus and delivered to the
  Experience Layer in real-time. Each notification is:
    – role-scoped (only sent to relevant role)
    – entity-linked (deep-link to the relevant screen)
    – actionable (where possible: action button in notification itself)

  Notification types by workspace:
    WS-01: DemandQuoted, DemandFulfilled, SlaWarning, SlaBreached
    WS-02: DemandCreated, DemandAccepted, SlaBreached, ChangeClosed
    WS-03: IncidentAssigned, IncidentResolved, ServiceRequestFulfilled/Rejected
    WS-04: IncidentCreated, SlaWarning, SlaBreached, KnownErrorWorkaroundInvalidated
    WS-05: ProblemCreated, RootCauseIdentified, KnownErrorWorkaroundInvalidated
    WS-06: ChangeRequested, ChangeApproved, EmergencyChangeActivated, TaskAssigned

8.3 Search

  Global search is scoped by role:
    SERVICE_CUSTOMER  → own demands + entitled catalog
    SERVICE_USER      → own incidents + requests + entitled catalog + KEDB
    SD_AGENT          → incidents + requests + KEDB (full tenant)
    PM_WORKER/LEADER  → problems + KEDB + incidents (read)
    CE_WORKER/LEADER  → changes + services + CI registry (full tenant)

  Search result types are labeled by entity type.
  Deep-link to the relevant screen on click.

8.4 AI Assistant (Stage: AI Assistant)

  When the AI Assistant stage is active, a natural language interface
  is available in the top navigation across all workspaces.

  The AI Assistant:
    – Translates intent into structured API calls
    – Assists or redirects (never decides autonomously)
    – Is role-scoped (can only surface and act on what the user is entitled to)

  Example interactions by workspace:
    WS-01: "What's the status of my demand for the new ERP module?"
           → GET /ops/demands?title=ERP — summary response
    WS-03: "My laptop won't connect to VPN since this morning."
           → KEDB check → if no match: pre-fills WS-03-002 form
    WS-04: "Show me all P1 incidents on IDENTITY_ACCESS services."
           → GET /ops/incidents?priority=P1&chapter=IDENTITY_ACCESS
    WS-06: "Which Technical Owners need to be involved if we change the SSO service?"
           → GET /catalog/services?name=SSO + dependency tree analysis → list Technical Owners
