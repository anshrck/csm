   197‚Üí## 3. Ticket State Machine Is Too Loose
   198‚Üí
   199‚ÜíEvidence:
   200‚Üí
   201‚ÜíTicket `PATCH` can directly set status to any supplied string. Dedicated routes exist for assign, resolve, close, reopen, but direct patch still allows broad transitions.
   202‚Üí
   203‚ÜíEnterprise issue:
   204‚Üí
   205‚ÜíEnterprise workflows need controlled transitions, required fields, ownership rules, and audit consistency.
   206‚Üí
   207‚ÜíPlan:
   208‚Üí
   209‚Üí1. Create `src/lib/ticket-state.ts`.
   210‚Üí2. Define allowed transitions:
   211‚Üí
   212‚Üí```ts
   213‚ÜíNEW -> TRIAGED
   214‚ÜíTRIAGED -> ASSIGNED
   215‚ÜíASSIGNED -> IN_PROGRESS
   216‚ÜíIN_PROGRESS -> WAITING_CUSTOMER
   217‚ÜíWAITING_CUSTOMER -> IN_PROGRESS
   218‚ÜíIN_PROGRESS -> RESOLVED
   219‚ÜíRESOLVED -> CLOSED
   220‚ÜíRESOLVED -> IN_PROGRESS // reopen before close
   221‚ÜíCLOSED -> IN_PROGRESS // reopen, permission restricted
   222‚Üí```
   223‚Üí
   224‚Üí3. Block direct status writes in `PATCH /api/tickets/[id]`.
   225‚Üí4. Require dedicated action routes for transitions.
   226‚Üí5. Enforce required fields:
   227‚Üí
   228‚Üí- assign requires user or group.
   229‚Üí- resolve requires resolutionCode and resolutionNotes.
   230‚Üí- close requires `RESOLVED` unless CM Leader override.
   231‚Üí- reopen requires reason.
   232‚Üí- waiting customer requires customer-visible comment.
   233‚Üí
   234‚ÜíAcceptance criteria:
   235‚Üí
   236‚Üí- Invalid ticket transitions return 409.
   237‚Üí- Every transition writes `TicketEvent` and `AuditLog`.
   238‚Üí- UI only exposes valid next actions.
