   203→## Routing design notes
   204→
   205→### URL ⇄ store feedback loop
   206→The catch-all page uses a `pushedRef` to break the cycle:
   207→1. User clicks sidebar nav → `navigate(view)` → store updates + navTick bumps.
   208→2. Catch-all's store→URL effect fires → `pushedRef = true` → `router.push(expectedPath)`.
   209→3. URL changes → Next.js re-renders with new `slug` prop.
   210→4. URL→store effect fires → sees `pushedRef === true` → clears ref + skips re-sync.
   211→
   212→Browser back/forward:
   213→1. URL changes (no `router.push` from us) → Next.js re-renders with new `slug`.
   214→2. URL→store effect fires → `pushedRef === false` → calls `syncFromUrl(slug)` → store updates.
   215→3. Store→URL effect fires → expected path already matches pathname → no push.
   216→
   217→This avoids infinite loops while keeping URL and store in sync.
   218→
   219→### Next.js 16 `params` is a Promise
   220→The catch-all page is a client component. In Next.js 16, `params` is a Promise and must be unwrapped with `React.use()` (or `await` in a server component) before accessing its properties. The page unwraps once at the top: `const { slug } = use(params);` — then uses `slug` everywhere. Caught this from the dev.log error on first test.
   221→
   222→### Detail views reuse list paths
   223→- `demand-detail` → `/demands/<id>`
   224→- `change-detail` → `/changes/<id>`
   225→- `ticket-detail` → `/tickets/<id>`
   226→
   227→`pathToView()` uses `DETAIL_VIEWS` to convert URL+id to the detail ViewKey; `viewToPath()` appends the id segment for the three detail ViewKeys.
