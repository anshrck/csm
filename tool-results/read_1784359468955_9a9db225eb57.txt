    25→    25→1. Purpose and Scope
    26→    26→────────────────────────────────────────────────────────────────────────────────
    27→    27→
    28→    28→1.1 Purpose
    29→    29→
    30→    30→This document specifies the Experience Layer of uSMS at two levels:
    31→    31→
    32→    32→  Level 1 — Functional Space Definition
    33→    33→    Per workspace: which entities are visible, which actions are available,
    34→    34→    which status views exist, which notifications are surfaced.
    35→    35→    Answers: "What belongs in this workspace?"
    36→    36→
    37→    37→  Level 2 — Screen Inventory & Interaction Model
    38→    38→    Per screen within each workspace: trigger conditions, data sources
    39→    39→    (from usms_ops and usms_catalog), available actions (state transitions),
    40→    40→    and role/entitlement constraints.
    41→    41→    Answers: "What happens on each screen?"
    42→    42→
    43→    43→This document does not specify visual design, component libraries, or
    44→    44→CSS implementation. It specifies functional behavior and data contracts.
    45→    45→Visual design follows in a subsequent Component Model (DAL-23).
    46→    46→
    47→    47→1.2 Scope — Six Workspaces
    48→    48→
    49→    49→  WS-01  CSM Portal             Service Customer
    50→    50→  WS-02  CSM Workspace          SCM Worker, CM Leader
    51→    51→  WS-03  Service Desk Portal    Service User
    52→    52→  WS-04  SD Workspace           SD Agent, SD Leader
    53→    53→  WS-05  PM Workspace           PM Worker, PM Leader
    54→    54→  WS-06  CE & Config Workspace  CE Worker, CE Leader, Technical Owner
    55→    55→
    56→    56→1.3 Governing Principles (inherited from DAL-20)
    57→    57→
    58→    58→  Consume Never Own
    59→    59→    No workspace stores or owns domain data. All data is consumed from
    60→    60→    uSMS domain APIs (/ops/*) and uSeC catalog APIs (/catalog/*).
    61→    61→
    62→    62→  Assist or Redirect
    63→    63→    Business operations (state transitions, approvals, closures) are
    64→    64→    executed by domain services — not by the Experience Layer.
    65→    65→    The workspace surfaces the action; the API enforces the rule.
    66→    66→
    67→    67→  Real-Time by Default
    68→    68→    All workspace state is event-driven. Lists, counters, and status
    69→    69→    indicators update via Event Bus subscription — no polling.
    70→    70→
    71→    71→  Entitlement-Governed Rendering
    72→    72→    Every screen, panel, and action button is rendered based on the
    73→    73→    caller's resolved entitlement and role. No UI configuration bypasses
    74→    74→    this. If the API would deny the action, the button does not appear.
    75→    75→
    76→    76→  Reduced to the Max (HRX)
    77→    77→    No workspace surface adds cognitive load without governance value.
    78→    78→    Every element must justify its presence by enabling a decision,
    79→    79→    surfacing a status, or triggering a legitimate action.
    80→    80→
    81→    81→1.4 Notation
    82→    82→
    83→    83→  Screen ID:    WS-{workspace}-{sequence}  (e.g. WS-01-003)
    84→    84→  Data:         [Entity] from schema        (e.g. [Demand] from usms_ops)
    85→    85→  Actions:      → STATE                    (e.g. → QUOTED)
    86→    86→  Constraints:  role, scope, status gate   (e.g. SERVICE_CUSTOMER only)

    90→    90→2. WS-01 — CSM Portal  (Service Customer)
    91→    91→────────────────────────────────────────────────────────────────────────────────
    92→    92→
    93→    93→2.1 Functional Space Definition
    94→    94→
    95→    95→  Purpose
    96→    96→    The CSM Portal is the Service Customer's strategic interface with the
    97→    97→    service organization. It surfaces the customer's demand portfolio,
    98→    98→    active service commitments, SLA performance, and the Service Catalog
    99→    99→    available to their organizational scope.
   100→   100→
   101→   101→  Role
   102→   102→    SERVICE_CUSTOMER — scoped to their OrgNode and children.
   103→   103→
   104→   104→  Visible Entities
   105→   105→    [Demand]             Own demands only (serviceCustomerId = caller)
   106→   106→    [Service]            Services in usms_catalog entitled to their OrgNode
   107→   107→    [ServiceOffering]    Entitled offerings within their scope
   108→   108→    [SLAProfile]         Profiles attached to their active services
   109→   109→    [SlaEvent]           SLA performance events for their services
   110→   110→    [Entitlement]        Their OrgNode's entitlement records (read-only)
   111→   111→
   112→   112→  Available Actions
   113→   113→    Submit new demand
   114→   114→    Accept or decline a quote
   115→   115→    Close a fulfilled or redirected demand
   116→   116→    View SLA performance per service
   117→   117→
   118→   118→  Status Views
   119→   119→    Demand pipeline: NEW / UNDER_REVIEW / QUOTED / ACCEPTED / IN_CHANGE / FULFILLED
   120→   120→    SLA health: per active service (green / warning / breached)
   121→   121→    Service Catalog: available offerings within entitlement scope
   122→   122→
   123→   123→  Notifications
   124→   124→    DemandQuoted          — quote ready for review
   125→   125→    DemandFulfilled       — demand completed
   126→   126→    DemandRejected        — demand not feasible (with reason)
   127→   127→    SlaWarning            — SLA at 70% threshold for an entitled service
   128→   128→    SlaBreached           — SLA breached for an entitled service
   129→   129→    ServiceCatalogUpdated — new service available within scope
   130→   130→
   131→   131→  Excluded
   132→   132→    Incident records (those belong to SD Portal)
   133→   133→    Problem records (internal process — not customer-facing)
   134→   134→    Change records (internal — customer sees effect via Demand status)
   135→   135→    Other customers' data (strict OrgNode scope)

   381→   381→    Redirect or reject demand

  1438→  1438→
  1439→  1439→  Trigger
  1440→  1440→    Row click from WS-06-002, dashboard action, or notification deep-link.
  1441→  1441→
  1442→  1442→  Purpose
  1443→  1443→    The most complex work surface in uSMS. Full change lifecycle management,
  1444→  1444→    service composition analysis, task coordination, and approval.
  1445→  1445→
  1446→  1446→  Data
  1447→  1447→    [Change]            Full record
  1448→  1448→    [Service]           Affected services — full catalog detail + composition tree
  1449→  1449→    [ServiceDependency] Dependency graph for impact analysis
  1450→  1450→    [Demand]            Origin demand (if originType = DEMAND) — status + customer context
  1451→  1451→    [Problem]           Origin problem (if originType = PROBLEM) — root cause + assessment
  1452→  1452→    [ProcessHandover]   Handover records referencing this change
  1453→  1453→
  1454→  1454→  Sections
  1455→  1455→
  1456→  1456→    Change Header
  1457→  1457→      Title / Type / Complexity / Status / Origin / CE Worker / CE Leader
  1458→  1458→
  1459→  1459→    Origin Context Panel
  1460→  1460→      If DEMAND: customer name / demand title / commitment notes / quote
  1461→  1461→      If PROBLEM: root cause description / impact assessment / PM decision
  1462→  1462→      If STANDARD: originating Service Request detail
  1463→  1463→
  1464→  1464→    Service Composition Panel  ← CE Worker's primary analysis tool
  1465→  1465→      "Affected Services & Composition"
  1466→  1466→      For each serviceId in affectedServiceIds:
  1467→  1467→        Service name / Layer / Chapter / Technical Owner
  1468→  1468→        Dependency tree (what does this service depend on?)
  1469→  1469→        Layer indicator: BUSINESS / APPLICATION / PLATFORM / INFRASTRUCTURE
  1470→  1470→      [Add Affected Service] → catalog search + add to affectedServiceIds
  1471→  1471→      [Remove Affected Service]
  1472→  1472→      Layer direction validation: warning if dependency rule would be violated.
  1473→  1473→
  1474→  1474→    Technical Owner Task Panel  (editable in PLANNING status)
  1475→  1475→      "Task Assignment"
  1476→  1476→      Per Technical Owner identified from composition analysis:
  1477→  1477→        Owner name / ServiceChapter / CI scope / Assigned task description
  1478→  1478→      technicalOwnerTasksJson built here.
  1479→  1479→      [Notify All Technical Owners] → task notifications sent
  1480→  1480→
  1481→  1481→    Implementation Plan Field
  1482→  1482→      implementationPlan — rich text, editable in PLANNING status.
  1483→  1483→      Template hints: rollback plan, test criteria, verification steps.
  1484→  1484→
  1485→  1485→    Approval Section (CE_LEADER — visible in PLANNING status)
  1486→  1486→      Complexity selector (if not yet set): SIMPLE / MEDIUM / COMPLEX

  1488→  1488→      For COMPLEX: Change Board session notes (approvalNotes)
  1489→  1489→
  1490→  1490→    Verification Section (CE_WORKER — visible in IMPLEMENTATION status)
  1491→  1491→      Task completion status from Technical Owners (per technicalOwnerTasksJson)
  1492→  1492→      verificationNotes field — "service functional, catalog updated"
  1493→  1493→      [Confirm Catalog Update] — sets catalogUpdatedAt (required for BR-CE-004)
  1494→  1494→
  1495→  1495→    Post-Implementation Review (EMERGENCY changes — visible after CLOSED)
  1496→  1496→      postImplementationReview field
  1497→  1497→      [Complete Post-Review] → sets isEmergencyPostReviewDue = FALSE
  1498→  1498→

  1500→  1500→

  1502→  1502→
  1503→  1503→    Status = REQUESTED  (CE_WORKER)
  1504→  1504→      [Assign to Me]      PATCH assignedCeWorkerId

  1506→  1506→                          BR-CE-001: affectedServiceIds must be set first

  1509→  1509→      [Set Complexity]    PATCH complexity (for NORMAL changes — BR-CE-002)

  1511→  1511→      POST /ops/changes/:id/plan → PLANNING
  1512→  1512→      Requires: implementationPlan, technicalOwnerTasksJson, complexity (NORMAL)
  1513→  1513→
  1514→  1514→    Status = PLANNING  (CE_LEADER)
  1515→  1515→      [Approve]     POST /ops/changes/:id/approve → APPROVED
  1516→  1516→                    SIMPLE: CE_LEADER alone
  1517→  1517→                    MEDIUM: CE_LEADER + approvalNotes referencing PO
  1518→  1518→                    COMPLEX: Change Board session required (BR-CE-003)
  1519→  1519→      [Reject]      POST /ops/changes/:id/reject → REJECTED
  1520→  1520→                    Requires: rejectionReason
  1521→  1521→
  1522→  1522→    Status = APPROVED  (CE_WORKER)
  1523→  1523→      [Start Implementation]  POST /ops/changes/:id/implement → IMPLEMENTATION
  1524→  1524→                              Notifies all Technical Owners
  1525→  1525→
  1526→  1526→    Status = IMPLEMENTATION  (CE_WORKER)
  1527→  1527→      [Complete Verification] POST /ops/changes/:id/verify → VERIFICATION
  1528→  1528→                              Requires: all Technical Owner tasks marked complete
  1529→  1529→                              verificationNotes set
  1530→  1530→
  1531→  1531→    Status = VERIFICATION  (CE_WORKER)
  1532→  1532→      [Close Change]    POST /ops/changes/:id/close → CLOSED
  1533→  1533→                        Requires: catalogUpdatedAt set (BR-CE-004)
  1534→  1534→                        Triggers: ServiceCatalogUpdated event
  1535→  1535→                        Triggers: Demand → FULFILLED (if originDemandId)
  1536→  1536→                        Triggers: Problem → CLOSED (if originProblemId)
  1537→  1537→                        Sets: isEmergencyPostReviewDue = TRUE (if EMERGENCY)
  1538→  1538→
  1539→  1539→    TECHNICAL_OWNER actions (any IMPLEMENTATION change with tasks assigned to them)
  1540→  1540→      [Mark Task Complete]  PATCH technicalOwnerTasksJson (task.status = DONE)
  1541→  1541→                            Emits task completion event to CE Worker
  1542→  1542→
  1543→  1543→    CE_LEADER — Emergency path (any status)
  1544→  1544→      [Declare Emergency]   Create new EMERGENCY change directly
  1545→  1545→                            POST /ops/changes (type = EMERGENCY)
  1546→  1546→                            Emits EmergencyChangeActivated
  1547→  1547→      [Expedited Approve]   Skip ASSESSMENT and PLANNING
  1548→  1548→                            POST /ops/changes/:id/approve directly from REQUESTED

  1693→  1693→

  1696→  1696→    – role-scoped (only sent to relevant role)
  1697→  1697→    – entity-linked (deep-link to the relevant screen)
  1698→  1698→    – actionable (where possible: action button in notification itself)

  1700→  1700→  Notification types by workspace:
  1701→  1701→    WS-01: DemandQuoted, DemandFulfilled, SlaWarning, SlaBreached
  1702→  1702→    WS-02: DemandCreated, DemandAccepted, SlaBreached, ChangeClosed
  1703→  1703→    WS-03: IncidentAssigned, IncidentResolved, ServiceRequestFulfilled/Rejected
  1704→  1704→    WS-04: IncidentCreated, SlaWarning, SlaBreached, KnownErrorWorkaroundInvalidated
  1705→  1705→    WS-05: ProblemCreated, RootCauseIdentified, KnownErrorWorkaroundInvalidated
  1706→  1706→    WS-06: ChangeRequested, ChangeApproved, EmergencyChangeActivated, TaskAssigned
  1707→  1707→
  1708→  1708→8.3 Search
  1709→  1709→
  1710→  1710→  Global search is scoped by role:
  1711→  1711→    SERVICE_CUSTOMER  → own demands + entitled catalog
  1712→  1712→    SERVICE_USER      → own incidents + requests + entitled catalog + KEDB
  1713→  1713→    SD_AGENT          → incidents + requests + KEDB (full tenant)
  1714→  1714→    PM_WORKER/LEADER  → problems + KEDB + incidents (read)
  1715→  1715→    CE_WORKER/LEADER  → changes + services + CI registry (full tenant)
  1716→  1716→

  1718→  1718→  Deep-link to the relevant screen on click.
  1719→  1719→
  1720→  1720→8.4 AI Assistant (Stage: AI Assistant)
  1721→  1721→
  1722→  1722→  When the AI Assistant stage is active, a natural language interface
  1723→  1723→  is available in the top navigation across all workspaces.
  1724→  1724→
  1725→  1725→  The AI Assistant:
  1726→  1726→    – Translates intent into structured API calls
  1727→  1727→    – Assists or redirects (never decides autonomously)
  1728→  1728→    – Is role-scoped (can only surface and act on what the user is entitled to)
  1729→  1729→

  1731→  1731→    WS-01: "What's the status of my demand for the new ERP module?"
